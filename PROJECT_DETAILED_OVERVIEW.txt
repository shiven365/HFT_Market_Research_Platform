PROJECT DETAILED OVERVIEW
Interactive Market Microstructure and HFT Simulation Platform

====================================================================
1) HIGH-LEVEL PURPOSE
====================================================================
This project is a BTCUSDT research and simulation platform with:
- A FastAPI backend that loads CSV data, computes analytics/features, and runs backtests.
- A React frontend that visualizes market structure, research insights, 3D liquidity views,
  and strategy simulation results.

Main idea:
- Historical data is preloaded in backend memory.
- Frontend pages call REST APIs through a single API client.
- Strategy runs are computed on demand and stored in-memory by backtest_id.

3) GLOBAL DATA FLOW: FILE TO FILE
====================================================================
A) Backend startup data flow:
1. backend/app/main.py -> startup_event()
2. startup_event() -> get_data_paths()
3. get_data_paths() resolves:
   - backend/Datasets/BTCUSDT-1m-2024-01.csv
   - backend/Datasets/BTCUSDT-trades-2024-01-sample-100k.csv
4. read_klines() and read_trades() parse CSV into Python dict lists.
5. Data stored in global memory:
   - KLINES
   - TRADES
   - SNAPSHOT_CACHE
   - BACKTEST_RESULTS

B) Frontend boot flow:
1. frontend/src/main.jsx mounts React app with BrowserRouter.
2. frontend/src/App.jsx maps URL paths to pages.
3. Each page either:
   - calls backend via frontend/src/api/client.js, or
   - loads static file frontend/public/market-snapshot.json (Home page).

C) API call flow pattern (shared):
1. Page function calls getMeta/getKlines/etc.
2. api/client.js -> buildUrl(path, params)
3. fetch() request to FastAPI (default API base: http://127.0.0.1:8000)
4. Backend endpoint computes response using in-memory KLINES/TRADES.
5. JSON response returned to page.
6. Page stores response in React state and passes it to chart components.

D) Backtest flow:
1. StrategyLab page submits POST /simulate.
2. Backend run_backtest() computes metrics/equity/trade log.
3. Result stored in BACKTEST_RESULTS[backtest_id].
4. Frontend navigates to /backtest/:id.
5. BacktestResults page calls GET /backtest/{id} and renders analytics.

4) BACKEND API DESIGN AND FORMULAS
====================================================================
File: backend/app/main.py

4.1 Utility/parsing
- parse_timestamp(value)
  Accepts epoch milliseconds (digits) or ISO datetime. Returns ms.

- sample_stdev(values)
  Sample standard deviation:
  variance = sum((x - mean)^2) / (n - 1)
  stdev = sqrt(variance)

4.2 Filtering and aggregation
- filter_klines(rows, start_ms, end_ms)
  Keeps candles where open_time is inside range.

- filter_trades(rows, start_ms, end_ms)
  Keeps trades where time is inside range.

- aggregate_klines(rows, interval)
  For 5m/15m/1h, merges 1m candles by bucket:
  bucket = open_time // bucket_ms
  open = first open in bucket
  high = max highs
  low = min lows
  close = last close
  volume = sum volume

4.3 Summary/statistics
- compute_summary(klines, trades)
  return_pct = ((last_close - first_close) / first_close) * 100
  minute_returns = (curr_close - prev_close) / prev_close
  volatility_pct = sample_stdev(minute_returns) * 100
  average_trade_size = sum(qty) / number_of_trades

- compute_returns_series(klines)
  return_pct per candle:
  ((curr_close - prev_close) / prev_close) * 100

- compute_rolling_volatility(returns_series, window)
  Uses rolling window of returns converted to decimal (return_pct / 100), then:
  rolling_volatility_pct = sample_stdev(window_returns_decimal) * 100

4.4 Histogram engine (used by Research Insights)
- build_histogram(values, bins_count, lower_q=None, upper_q=None, log_scale=False)
  Pipeline:
  1) Convert to float list.
  2) Optional log transform: v -> log10(v) for v > 0.
  3) Optional quantile clipping:
     lo = quantile(sorted_values, lower_q)
     hi = quantile(sorted_values, upper_q)
     each value clipped into [lo, hi]
  4) Equal-width bins in transformed domain.
  5) Bin assignment: idx = int((value - min_value) / width)
  6) If log_scale used, convert bin_start/bin_end/mid back via 10^x.

- quantile(sorted_values, q)
  Linear interpolation between nearest ranks.

4.5 Buyer vs seller side classification
- buyer_seller_stats(trades)
  seller_initiated = count(is_buyer_maker == True)
  buyer_initiated = total - seller_initiated
  percentages computed over total.

4.6 Snapshot frame engine (for 3D Order Book view)
- build_snapshot_frames(trades, frame_minutes, price_bucket)
  frame_key = trade.time // frame_ms
  price_bucket_value = floor(price / price_bucket) * price_bucket
  per (frame_key, price_bucket_value):
    accumulate volume += qty
    accumulate trade_count += 1

- /snapshot endpoint enriches each bucket with:
  mode: volume or trade_count
  activity = bucket[mode]
  intensity = activity / max_activity_in_selected_frame

4.7 Strategy signal logic
- strategy_direction(strategy, klines, i, threshold_pct, lookback)
  threshold = threshold_pct / 100
  one_bar_return = (curr_close - prev_close) / prev_close

  momentum:
  - if one_bar_return >= threshold: LONG (+1)
  - if one_bar_return <= -threshold: SHORT (-1)

  mean_reversion:
  - if one_bar_return >= threshold: SHORT (-1)
  - if one_bar_return <= -threshold: LONG (+1)

  volume_spike:
  - avg_volume over lookback bars
  - spike condition: current_volume >= avg_volume * (1 + threshold)
  - direction by candle body:
    close > open -> LONG
    close < open -> SHORT

4.8 Backtest execution formulas
- run_backtest(klines, request)
  Core assumptions:
  - Notional per trade = 1000 USD
  - Entry/exit at next/open bars with slippage
  - Round-trip fees = 2 * fee_rate

  fee_rate = fee_bps / 10000
  slippage_rate = slippage_bps / 10000

  LONG trade:
  entry_price = entry_open * (1 + slippage_rate)
  exit_price  = exit_open  * (1 - slippage_rate)
  gross_return = (exit_price - entry_price) / entry_price

  SHORT trade:
  entry_price = entry_open * (1 - slippage_rate)
  exit_price  = exit_open  * (1 + slippage_rate)
  gross_return = (entry_price - exit_price) / entry_price

  net_return = gross_return - 2 * fee_rate
  pnl_usd = net_return * 1000

  equity update:
  equity = equity * max(0.0001, 1 + net_return)

  metrics:
  - total_return_pct = (equity - 1) * 100
  - win_rate_pct = wins / trade_count * 100
  - avg_trade_return_pct = mean(trade_returns) * 100
  - max_drawdown_pct = compute_max_drawdown(equity_curve) * 100
  - sharpe_like = (avg_trade / stdev(trade_returns)) * sqrt(trade_count)

- compute_max_drawdown(equity_points)
  peak tracks highest equity seen so far.
  drawdown at point = (peak - equity) / peak
  max_drawdown = max drawdown over curve.

4.9 Main endpoints and usage
- GET /meta
  Data availability range for klines/trades.

- GET /summary
  High-level summary stats for selected window.

- GET /klines
  Candle series with interval aggregation and limit.

- GET /trades/sample
  Most recent trade rows for table.

- GET /snapshot
  3D bucket data for selected frame/mode.

- GET /features
  Research feature payload:
  - rolling volatility series
  - trade size histogram (log-scaled + upper clipped at 99.5th percentile)
  - return distribution (clipped to central 98%, 1st to 99th percentile)
  - buyer/seller ratio

- GET /insights
  Aggregated research statistics + natural-language observations.

- POST /simulate
  Runs strategy backtest; returns backtest_id + metrics.

- GET /backtest/{id}
  Returns saved simulation payload from in-memory store.

6) PAGE-BY-PAGE FUNCTIONALITY + FORMULAS

6.1 Home page
Files:
- frontend/src/pages/Home.jsx
- frontend/src/components/CandleMarketBoard.jsx
- frontend/src/components/MarketScene3D.jsx
- frontend/public/market-snapshot.json
- backend/scripts/generate_market_snapshot.py

Data source and flow:
1. Home.jsx fetches /market-snapshot.json (static public file).
2. If fetch fails, fallbackSnapshot object is used.
3. Data mapped into ticker cards, market board, flow list, signal list.

How market-snapshot.json is generated:
- Script reads both CSV files.
- Computes 1m, 15m, 1h, 24h percentage changes via:
  pct_change = ((new - old) / old) * 100
- 24h high/low/volume from last 1440 one-minute candles.
- 1H volatility from stdev of recent returns.
- Buy/Sell pressure from quote_qty split by is_buyer_maker.
- Produces latest 140 candles for chart section.

Chart formula (CandleMarketBoard):
- minPrice = min(low), maxPrice = max(high)
- range = maxPrice - minPrice
- toY(price) = padTop + ((maxPrice - price) / range) * drawableHeight
- Candle body direction:
  close >= open => up candle
  close < open => down candle

3D hero animation formula (MarketScene3D):
- scaleY = 0.35 + (sin(t*2 + phase) + 1) * 0.45
- group rotation continuously increments each frame.


6.2 Explorer page
Files:
- frontend/src/pages/Explorer.jsx
- frontend/src/components/CandleMarketBoard.jsx
- frontend/src/components/VolumeBarsChart.jsx
- frontend/src/api/client.js

Data source and flow:
1. Explorer calls GET /meta on mount to prefill date range inputs.
2. On date/timeframe change, loadData() runs Promise.all:
   - GET /summary
   - GET /klines?interval=...&limit=480
   - GET /trades/sample?limit=120
3. Responses stored into summary/candles/trades state.
4. Candles passed to CandleMarketBoard and VolumeBarsChart.
5. Trades shown in table (side inferred from is_buyer_maker).

Displayed formulas:
- Return card: signed display of backend return_pct.
- Volatility card: backend volatility_pct.
- VolumeBarsChart normalization:
  bar_height = (candle.volume / maxVolume) * chartHeight
- Trade side mapping:
  is_buyer_maker true -> Sell
  false -> Buy


6.3 Research Insights page
Files:
- frontend/src/pages/ResearchInsights.jsx
- frontend/src/components/DatasetSummary.jsx
- frontend/src/components/VolatilityChart.jsx
- frontend/src/components/TradeSizeHistogram.jsx
- frontend/src/components/ReturnDistributionChart.jsx
- frontend/src/components/BuyerMakerRatioChart.jsx
- frontend/src/components/ObservationsPanel.jsx

Data source and flow:
1. On mount, Promise.all requests:
   - GET /features?volatility_window=30&histogram_bins=28
   - GET /insights
2. DatasetSummary uses /insights metadata.
3. Charts render using /features payload.
4. ObservationsPanel renders narrative strings from /insights.observations.

Main formulas behind visuals:
- Rolling Volatility (backend): sample stdev of rolling returns window.
- Trade Size Histogram (backend):
  log10 transform on qty, upper quantile clipping at 99.5%, equal-width bins,
  then inverse 10^x for final labels.
- Return Distribution Histogram (backend):
  central clipping between 1st and 99th percentiles before binning.
- Buyer/Seller Ratio (backend):
  buyer_pct = buyer_initiated / total * 100
  seller_pct = seller_initiated / total * 100

Frontend rendering choices:
- Both histogram charts use logarithmic Y-axis for count visibility across skewed data.
- Non-zero bars are emphasized with minPointSize to avoid visually disappearing bins.


6.4 3D Order Book page
Files:
- frontend/src/pages/OrderBook3D.jsx
- frontend/src/components/OrderBookScene3D.jsx
- frontend/src/api/client.js

Data source and flow:
1. Page loads GET /snapshot with controls:
   mode, frame_index, frame_minutes, price_bucket, max_levels
2. Receives selected frame + bins + per-bin intensity.
3. Passes bins to OrderBookScene3D.
4. Hover callbacks provide selected bucket info panel.

Backend formulas used:
- Price bucketing:
  price_bucket_value = floor(price / price_bucket) * price_bucket
- Activity (mode dependent):
  activity = volume OR trade_count
- Intensity normalization:
  intensity = activity / max_activity_of_frame

3D rendering formulas (OrderBookScene3D):
- baseHeight = 0.35 + intensity * 5.4
- x spacing depends on bin count; bars centered around zero.
- z layering:
  z = (round(intensity * 4) - 2) * 0.58
- Animation pulse:
  pulse = 1 + sin(time*2.4 + idx*0.17) * 0.06
  rendered_height = baseHeight * pulse

====================================================================
8) IMPORTANT ASSUMPTIONS AND LIMITATIONS
====================================================================
- Backend storage is in-memory only:
  BACKTEST_RESULTS resets when server restarts.
- Data universe is the loaded CSV range only.
- Backtest execution model is simplified:
  - next-open fills
  - fixed slippage bps
  - fixed fee bps
  - fixed notional per trade
- Snapshot and feature analytics are deterministic on the loaded dataset.
